# Product Requirement Document (PRD): QRDoc (VitalPass)

## 1. 프로젝트 개요 (Project Overview)
* **프로젝트명**: 큐알닥 (QRDoc) / 바이탈패스 (VitalPass) [가칭]
* **서비스 형태**: 환자용 하이브리드 앱 (Flutter 기반 iOS & Android) + 의료진용 휘발성 웹 뷰어 (Responsive Web)
* **기술 키워드**: `Flutter`, `Google Gemini API (Multimodal OCR)`, `Node.js`, `Redis`, `Zero-Knowledge AES-256 Encryption`
* **한 줄 정의**: 
  > "Gemini AI로 처방전을 찍어 내 폰에 안전하게 보관하고, 병원 방문 시 일회성 QR 코드로 의료진에게 개인정보 유출 없이 빠르게 전달하는 스마트 건강 정보 지갑"

---

## 2. 제품의 목표 및 비전 (Goals & Vision)
* **보안성 극대화 (Zero-Knowledge)**: 환자의 민감한 의료 데이터는 서버에 절대 영구 저장되지 않으며, 환자의 디바이스 내부 보안 스토리지(Secure Storage)에만 암호화되어 보관됩니다.
* **입력 프로세스 단순화 (AI-First)**: 기존의 번거로운 약물 정보 수동 입력을 개선하여, Gemini API를 통해 처방전 사진 한 장으로 처방 내역을 완벽히 구조화합니다.
* **의료진 편의성 향상 (Zero-Install)**: 바쁜 의료진(의사/간호사)이 별도의 프로그램 설치나 가입 없이, 기본 스마트폰/태블릿 카메라로 QR 코드를 스캔해 즉시 웹 브라우저에서 안전하게 데이터를 확인할 수 있도록 설계합니다.

---

## 3. 핵심 타겟 사용자 및 사용 시나리오 (User Scenarios)

### 3.1 타겟 사용자
* **환자 (Primary User)**:
  * 여러 병원을 정기적으로 방문하여 복약 이력 관리가 필요한 만성질환자.
  * 새로운 병원에 갈 때마다 기존 진료 이력 및 알레르기 정보를 일일이 구두로 설명해야 하는 환자.
* **의료진 (Secondary User)**:
  * 초진 환자의 과거 약물 복용 이력이나 지병, 알레르기 상태를 빠르고 정확하게 확인해야 하는 의사 및 간호사.

### 3.2 핵심 사용 시나리오 (User Flow)
1. **처방전 등록 (환자)**:
   * 환자는 약국에서 처방전이나 약봉투를 받아 Flutter 앱의 카메라 기능으로 촬영합니다.
   * 앱은 이미지 데이터를 **중계 서버를 통해 안전하게 Gemini API로 프록시 요청**합니다. Gemini는 이미지 안의 복잡한 텍스트 문맥을 분석하여 `[약물명, 1회 복용량, 1일 복용 횟수, 총 복용 일수]`를 깔끔한 JSON 데이터로 추출합니다.
   * 환자는 화면에 뜬 AI의 인식 결과를 확인하고 최종 승인(또는 미세 조정)하여 내 폰 내부 안전 저장소에 저장합니다.
2. **의료진 공유용 QR 생성 (환자)**:
   * 병원 대기실 혹은 진료실에서 환자가 앱의 "공유용 QR 생성" 버튼을 터치합니다.
   * 앱 내부에서 환자의 건강 프로필과 복약 이력이 고유한 임시 대칭키(`Secret_Key`)로 암호화(`AES-256-GCM`)됩니다.
   * 암호화된 데이터 덩어리는 중계 서버(Node.js)로 전송되어 인메모리 DB(Redis)에 임시 저장(3분 TTL)되고, 고유한 데이터 ID(`Data_ID`)를 발급받습니다.
   * 앱 화면에는 복호화 키가 포함된 일회성 임시 웹 주소 기반의 QR 코드가 표시됩니다.
   * 예시 주소 구조: `https://viewer.qrdoc.com/view/{Data_ID}#{Secret_Key}`
3. **데이터 조회 및 휘발 (의료진)**:
   * 의사가 병원 태블릿이나 스마트폰 카메라로 QR 코드를 스캔합니다.
   * 의사의 브라우저에서 `Data_ID`에 해당하는 암호화된 데이터를 다운로드받고, URL의 `#` 뒤에 숨겨진 `Secret_Key`를 사용해 브라우저 메모리상에서 데이터를 복호화하여 화면에 표출합니다.
   * **[보안 보완] 일회성 조회 처리 (Burn After Reading)**: 중계 서버는 데이터를 조회하는 최초 1회 API 호출 시 즉시 Redis에서 데이터를 삭제하여 재전송 공격을 원천 방지합니다. 
   * 데이터 조회 후 3분이 경과하거나 화면을 닫으면, 브라우저 메모리는 지워지고 서버의 임시 데이터도 만료되어 완전히 영구 파기됩니다.

| **W-03** | 타임아웃 보호 | 의사용 화면이 열린 뒤 3분이 지나면 세션을 만료하고 화면을 강제 초기화하여 환자 데이터 노출 방지. | High |

## 5. 비기능적 요구사항 (Non-Functional Requirements)

### 5.1 보안 및 개인정보 (Security)
* **Zero-Knowledge Architecture**: 중계 서버는 오직 암호화된 바이너리 덩어리만 보관하며, 복호화용 암호키는 환자의 QR 코드 주소 내 `#` 뒤(Fragment Identifier)에 숨겨져 전송되므로 서버 개발자나 데이터베이스 해킹을 통해서도 환자 식별 정보를 읽을 수 없습니다.
* **로컬 무결성**: Flutter 앱은 모바일 OS 수준의 보안 영역(Keychain/Keystore)을 활용하여 로컬 DB의 복호화 키를 관리합니다.
* **API Key 보안**: 어떠한 외부 연동 Key도 클라이언트 앱 내에 하드코딩되지 않으며, 서버사이드 환경 변수로 안전하게 제어됩니다.

### 5.2 성능 및 사용성 (Performance & Usability)
* **Gemini 파싱 속도**: 이미지를 업로드하고 복약 일정이 앱에 구조화되어 표출되는 전체 처리 시간이 3초 이내여야 합니다.
* **의료진 조회 응답성**: 의사가 QR을 인식하고 브라우저에서 암호화 데이터를 서버로부터 받아와 복호화 후 화면을 렌더링하는 속도는 1.5초 이내여야 합니다.
* **고령자 최적화 UI**: 환자용 앱의 주 사용 타겟층인 중장년층/고령층을 위해 텍스트 크기 조절 옵션을 제공하고 버튼 가시성을 높입니다.
